# Interactive Tool Use ‚Äî Detection, Mode, and Model-Size Routing

Goal: Detect when a task involves an interactive script (local or remote), then route model interaction through persistent stdin/stdout sessions. For large models (>20B) expose session tools directly. For small models (‚â§14B) hide the session-management plumbing and let the harness drive the session lifecycle, while the model still sees prompts and supplies answers in natural language.

Status: planning

## 0. Architecture orientation (READ FIRST ‚Äî corrects the original draft)

This plan touches **two distinct packages** that the original draft conflated:

| Layer | Path | Role ||---|---|---|| **Runtime harness** | `src/smallctl/` | The production agent. Tool registry, dispatcher, mode/task classification, prompts, guards, FAMA tool policy. This is where the feature must live to work in production. || **Eval harness / optimizer** | `aho/` | Wraps the runtime for benchmarking. Owns the `ToolInterceptor` *instances* (`ProgressGuard`, `ArtifactDedupeInterceptor`), `aho/setup.py::create_interceptors`, `aho/harness_config.json`, `aho/challenges_30.json`, `aho/eval.py`. |

Key consequences the draft got wrong:

  • The `ToolInterceptor` **protocol** and `PipelineDispatcher` live in `src/smallctl/tools/dispatcher.py`, but **no interceptors are instantiated inside `src/smallctl`** ‚Äî they are all registered in `aho/setup.py::create_interceptors`. A transparent-mode interceptor that must work in production therefore **cannot** be modeled on `ProgressGuard`/`ArtifactDedupeInterceptor` (those are eval-only). Transparent driving belongs in the runtime: in `shell.py`'s foreground branch + a `graph/` guard + session state on `LoopState`.
  • `graph/progress_guard.py` is a module of **functions**, not a `ProgressGuard` class. The class named `ProgressGuard` is `aho/progress_guard.py`. A new interactive guard in the runtime should follow the **functions** convention used in `graph/`.
  • Mode-specific system-prompt text is assembled in `src/smallctl/prompts.py::build_system_prompt` (fragments from `prompt_fragments.py`), **not** `prompt_builder.py` (which only orchestrates retrieval/compaction).
  • `aho/eval.py::score_trial` reads only `expected_keywords` and `required_tool_calls` from `ground_truth`. The existing `required_tool_counts`/`required_tool_sequence` fields are **already ignored**, so any new `ground_truth` field is dead until `score_trial` is extended.

    ## 1. Detection ‚Äî extend, don't replace

    **Do NOT create a brand-new `interactive_detector.py` from scratch.** A keyword classifier (`task_involves_interactive_program`), an output-prompt detector (`detect_interactive_prompt`), and an installer probe (`is_interactive`) already exist. The work is to (a) widen `task_involves_interactive_program` into a richer classifier that returns *which transport* is needed, and (b) wire its result into the visibility flag.

    **New file:** `src/smallctl/harness/interactive_detector.py` ‚Äî a thin fa√ßade that composes the three existing detectors.

    Signals (any single strong signal or 2+ medium signals ‚Üí interactive):

    | Signal | Weight | Source ||---|---|---|| **GUI/TUI/game keywords** | strong | reuse `task_involves_interactive_program` marker set || **Explicit interactive keywords** | strong | extend marker set: "stdin", "user input", "prompt for", "REPL", "menu-driven", "choose option" || **CLI dialog markers** | medium | "yes/no", "Y/N", "press enter", "type your", "what is your" || **Server/daemon** | medium | "listen on port", "socket server", "chat server", "accept connection" || **Installer markers** | medium | reuse `is_interactive` probe result already computed in `shell.py`/`network.py` || **History ‚Äî output matches a prompt** | strong | `detect_interactive_prompt()` on the last tool output || **History ‚Äî `shell_exec` timed out with partial output** | medium | foreground exec hit `timeout_sec` after emitting bytes (likely blocked on stdin) |

    Output:

      def classify_interactive_need(task: str, state: LoopState) -> InteractiveNeed | None:      """None | InteractiveNeed(transport='local'|'remote'|'hybrid',                                mode='scripted'|'reactive', confidence=float)."""

    Two axes matter (see ¬ß4b): **transport** (where the process runs) and **mode** ‚Äî `scripted` (all inputs known up front, e.g. an installer answer sequence) vs `reactive` (each input depends on the program's last output, e.g. guess-the-number). This distinction is the crux of small-model handling.

    Placement in pipeline (corrected to real call sites):

    1. `harness/task_classifier.py::classify_task_mode()` ‚Äî add an `interactive_execute` label to the `_TASK_CLASSIFICATION_RULES` table (alongside `local_execute`, `hybrid_execute`, `remote_execute`). **This is the right hook ‚Äî NOT a new run-mode** (see ¬ß3).

    2. `tools/shell.py` foreground branch and `tools/network.py` ‚Äî call it (or reuse the already-computed `is_interactive` probe) to set `_expose_interactive_session_tools` for local sessions, mirroring the existing SSH path.

    3. `tools/control_task_complete_gates.py::task_complete_gate_interactive_program()` ‚Äî already exists; gate stays, detector feeds it.

    ## 3. Mode & tool visibility ‚Äî NOT a new run-mode

    The draft proposed adding `"interactive"` to `ModeDecisionService.decide()`. **This is wrong and high-cost.** `decide()` only ever returns `"tool_plan" | "chat" | "loop" | "planning"`, and `normalize_mode_decision()` collapses the model's vote to just `loop`/`chat`. Adding a fifth run-mode would ripple through every consumer of run-mode.

    Interactivity is a **task attribute**, not a run-mode. Model it the same way the codebase already models "remote vs local": via `classify_task_mode()` labels + the FAMA tool-visibility flag. Interactive tasks still run in the existing `loop` run-mode.

It composes naturally with the existing `local_execute` / `remote_execute` / `hybrid_execute` labels ‚Äî store the transport on `state` separately so the gate and prompt builder can read it.

    ## 4. Model-Size Routing ‚Äî THE KEY TWO-FOLD APPROACH

    The central design: **large models get raw tool access; small models get transparent harness-driven interaction.**

    4a. Large models (>20B, `guards.py::is_over_twenty_b_model_name`, threshold size > 20.0)

    **Expose interactive tools directly.** The model sees `interactive_start/send/read/close` and `ssh_session_start/send/read/close` (un-hidden via the ¬ß3b flag).

    System-prompt section, added as a fragment in `prompts.py::build_system_prompt` (gated on the `interactive_execute` task mode), **not** `prompt_builder.py`:

    [INTERACTIVE SCRIPT MODE]

    This task involves an interactive script. Workflow:

    1. Write the script with write_file or file_patch

    2. [Local] interactive_start [Remote] ssh_file_write then ssh_session_start

    3. Send input with interactive_send / ssh_session_send

    4. Read output with interactive_read / ssh_session_read

    5. Iterate: fix bugs, restart, re-interact

    6. Call task_complete when the script behaves correctly

    No extra harness logic ‚Äî the model orchestrates the full lifecycle. This is the thin path.

    4b. Small models (‚â§14B, `guards.py::is_small_model_name`, threshold size ‚â§ 14.0)

    **Session-management tools stay hidden. The harness drives the lifecycle.** But be precise about what "transparent" can mean:

    **You cannot hide the I/O content from the model.** For a guess-the-number game the model *must* see each prompt ("Too low!") to choose the next input. What you hide is the **session plumbing** (start/send/read/close), not the prompts/answers. "Transparent" = the model never issues a session tool call; it reacts to output and supplies the next answer in its normal turn.

    This splits into two sub-strategies, chosen by the detector's `mode` axis (¬ß1):

    # Strategy A ‚Äî `scripted` interaction (inputs known up front)

    Best for installers, fixed quizzes, "answer yes to all". Expose **one** high-level tool the small model *can* reliably call:

      interactive_run(*, command, inputs: list[str], cwd=None, state=None, harness=None)    ‚Üí {status, transcript, exit_code, prompts_seen}

    The harness starts the session, sends each line from `inputs` when a prompt is detected (`detect_interactive_prompt`), and returns the full transcript in one envelope. The model plans the whole interaction in a single call ‚Äî no turn-by-turn state machine, far more reliable for small models. This subsumes the existing `yes |` installer guard.

    # Strategy B ‚Äî `reactive` interaction (input depends on output)

    Best for games/REPLs. Here the harness manages the session across turns while the model reacts. The flow, grounded in real hooks:

    1. Model calls `shell_exec(command="python3 quiz.py")` (its normal behavior; it never sees session tools).

    2. **`shell.py` foreground branch** (`shell_exec_foreground`, currently spawns with `stdin=DEVNULL`) detects interactivity ‚Äî process emits bytes then blocks past a short idle window, **or** `detect_interactive_prompt()` matches partial output. Instead of returning a one-shot result, it transparently calls `local_interactive_sessions.interactive_start` (stdin=PIPE), stores the `session_id` on `LoopState`, and returns a `ToolEnvelope(success=True, status="awaiting_input", output=, metadata={"interactive_session": True, "transparent": True})`.

    3. A new `graph/` guard (functions, mirroring `graph/progress_guard.py`) renders the active session into the next prompt and sets the gate: **while a transparent session is `AWAITING_INPUT`, the model's next turn is interpreted as the input.**

    4. Disambiguation is the hard part. Two layers:

  • Extend `classify_runtime_intent()` (`task_classifier.py`, returns `RuntimeIntent(label, task_mode)`) with a label `"interactive_input"`, recognized **only** when `state` has an awaiting session. Strip a leading directive verb ("send/type/answer/enter X") if present; otherwise treat the whole assistant message content as the literal input.
  • If the model emits a tool call instead of text, intercept `shell_exec`/any tool while a session awaits input and re-route or nudge.

    • 5. Harness calls `interactive_send(session_id, input)` behind the scenes, returns new output as the next tool result.

    6. Loop until the process exits (`exit_code` set) or `task_complete`. On exit, auto-close the session.

    # Reliability guards (small models are the failure-prone path)

  • **Bounded turns:** cap reactive sends per session (e.g. 12); on overflow, fall back (below).
  • **Confidence fallback to exposed:** if the detector's confidence is low, or the model keeps emitting non-input turns while a session awaits, inject:

    • `"I need explicit input. Reply with just the value to send, or describe the full sequence of answers."`

    For >14B-capable mid models, fall back to the exposed tools instead.

  • **No silent input:** every auto-`interactive_send` is logged via `RunLogger` so eval/replay can see what the harness injected on the model's behalf.

    • # Why this is safer than the original draft

    The original ¬ß4b assumed the harness could parse "Send 'Paris'" out of free-form text on every turn. That is brittle for small models. The corrections: (a) prefer the single-call `interactive_run` for scripted cases (the common installer case), (b) only arm the natural-language interpreter while a session is provably `AWAITING_INPUT`, (c) treat the *entire* assistant turn as input when no directive verb is found, and (d) keep a hard fallback to the exposed tools.

    4c. Mid-size models (14B < size ‚â§ 20B)

    Neither `is_small_model_name` (‚â§14) nor `is_over_twenty_b_model_name` (>20) matches this band ‚Äî confirm the gap in `guards.py`. Default these to the **exposed** path (Strategy of 4a) since they generally handle multi-tool sequences; allow override via config:

    "interactive_tool_policy": "auto" | "expose" | "transparent"

    Add this key to `SmallctlConfig` (`src/smallctl/config.py`) so `resolve_config` accepts it (unknown keys are filtered/warned at lines 341-346), and to `HarnessConfig` (`src/smallctl/harness/config.py`). For eval runs, also surface it in `aho/harness_config.json`.

    ## 5. Interaction runtime semantics ‚Äî snapshots, output modes, waiting, timeouts

    Both paths (exposed for large, harness-driven for small) sit on one read/snapshot/timeout layer. The SSH module already implements the snapshot model (`network_interactive_sessions.py::_interactive_session_snapshot`); the local twin must match it, and **both should gain the features below** ‚Äî the SSH module currently stores `timeout_sec`/`started_at` but **never enforces them**, appends stdout to a list with **no cap**, and reads with `idle_timeout_sec=None`. Fix these while building the local module so the two stay symmetric.

    5a. Snapshots & status ‚Äî the MENU case

    A snapshot is what the model uses to "see what the script needs". Today `_interactive_session_snapshot()` returns `{status, detected_prompt, stdout_tail, stderr_tail, output_tail, exit_code}` and derives status as `exited` / `waiting_for_input` (when `detect_interactive_prompt` matches the tail) / `running`. `ssh_session_read` / `interactive_read` *is* the snapshot tool.

    For menus (`detect_interactive_prompt` already classifies `choice` prompts via `Choice:\s*\[(?P[^\]]+)\]`, e.g. `Choice: [1-4]`):

  • **Capture the whole menu, not just the prompt line.** A menu prints many option lines then a short prompt; snapshots return only a tail. Add a per-session **read offset** and return `new_output` (everything since the last read) alongside `output_tail`, and raise the local default `max_chars`, so the model reliably sees the full option list once without re-reading scrollback.
  • **Extend the status enum** beyond the current three:

    • `running` ¬∑ `streaming` (busy + new output flowing) ¬∑ `waiting_for_input` ¬∑ `idle` (no output, no prompt, not exited ‚Äî possibly an *unrecognized* prompt) ¬∑ `exited` ¬∑ `timed_out`.

  • Snapshot gains `new_output`, `bytes_since_last_read`, `total_bytes`, `last_output_at`, `elapsed_sec`.

    • Exposed (large): model reads the menu, calls `interactive_send(input="2")`.

    Transparent (small): the menu text is rendered into the prompt; the model's next turn (reactive) or next queued `interactive_run` input (scripted) is the selection.

    5b. Output modes ‚Äî prompt vs menu vs streaming logs vs progress bars ‚Äî the LOGS case

    Classify each snapshot's `new_output` before showing the model, so a build/log stream isn't mistaken for an input prompt:

    | Output mode | Heuristic | Handling ||---|---|---|| **Prompt / menu** | `detect_interactive_prompt()` matches the tail | `waiting_for_input` ‚Üí solicit input || **Streaming logs** | new bytes arriving, no prompt match, multiple `\n` | `streaming` ‚Üí tail/summarize, do NOT solicit input, keep polling || **Progress bar** | new output dominated by `\r` with no `\n` | collapse `\r` segments to the last frame before tailing; `streaming` || **Quiet / idle** | no new bytes for `idle_grace_sec`, not exited | `idle` ‚Üí likely an unrecognized prompt; transparent mode nudges/falls back |

  • **Cap the in-memory buffer.** The SSH module appends to `session["stdout"]` unbounded ‚Äî a log spewer can OOM. Add a ring-buffer cap (e.g. 256 KB; reuse `process_lifecycle.py::truncate_output`), keep the tail, track `total_bytes`.
  • **Small models must not be flooded:** in transparent mode, compact stream output (last N lines + `[‚Ä¶ X lines of log ‚Ä¶]`) before it reaches the prompt.

    • 5c. Waiting on a running session (local & remote) ‚Äî the WAITING case

    Reads should block until something meaningful happens instead of fixed-sleep polling. Extend the read tools (identical semantics local & SSH; only transport differs):

      interactive_read(*, session_id, wait_sec=0.0,                   until="now"|"prompt"|"exit"|"change",                   deadline_sec=30.0, max_chars=6000)

  • `until="now"` ‚Äî current behavior (sleep `wait_sec`, snapshot).
  • `until="prompt"` ‚Äî return as soon as a prompt is detected, the process exits, or `deadline_sec` elapses.
  • `until="exit"` ‚Äî return when the process exits or `deadline_sec` elapses ("run and wait for it to finish").
  • `until="change"` ‚Äî return on the next new output or status change.

    • Implement as an internal poll loop (`asyncio.sleep(~0.1)` over the background-collected buffers) bounded by `deadline_sec`.

  • **Exposed (large):** model picks `until` explicitly.
  • **Transparent (small):** the harness chooses `until` from the detector mode (`reactive` ‚Üí `prompt`; "run to completion" step ‚Üí `exit`) and renders the resulting snapshot. The small model never sees the polling.

    • 5d. Timer / timeout handling

    Three independent timers ‚Äî the SSH module currently enforces **none**; implement in both modules:

    | Timer | Default | Trigger | Action ||---|---|---|---|| **Per-read `deadline_sec`** | 30 s (matches the existing `wait_sec` clamp) | one read call exceeds it | return current snapshot, status unchanged (`running`/`streaming`) ‚Äî **not** an error; caller may read again || **Idle timeout `idle_grace_sec`** | 20 s | running, but no new bytes and no prompt | status `idle`; transparent nudges/falls back; exposed surfaces `idle` so the model decides (send newline? close?) || **Session wall-clock `timeout_sec`** | 900 s (already stored) | `elapsed_sec > timeout_sec` | auto `stop_process` + cleanup, return `status="timed_out"`, `metadata.reason="session_timeout"` |

  • Enforce wall-clock on every read/send: `elapsed = time.time() - session["started_at"]`; if exceeded, terminate via `stop_process(proc, harness=..., timeout=2.0)` then `_cleanup_interactive_session`.
  • The transparent reactive loop keeps its own ceilings (max sends/turns, ¬ß4b) layered on these wall-clock timers so a stuck game can't burn the whole step budget.
  • The `task_complete` gate (¬ß6b) rejects completion when any session ended `timed_out` or `idle` rather than `exited` ‚Äî a timeout is not success.
  • Emit a `UIEvent` ALERT on `timed_out` (mirror `graph/progress_guard.py::_check_completion_confabulation`'s nudge style).

    • ## 6. Interaction Lifecycle Guards

    6a. Interactive progress guard (new ‚Äî functions, not a class)

    `graph/progress_guard.py` is a **module of functions** (`_check_progress_stagnation`, `_check_completion_confabulation`, etc.) that inject recovery nudges as `ConversationMessage(role="user", metadata={"is_recovery_nudge": True, "recovery_kind": ...})` and emit `UIEvent`s. The `ProgressGuard` *class* is eval-only (`aho/progress_guard.py`) ‚Äî do **not** model the runtime guard on it.

    **New file:** `src/smallctl/graph/interactive_progress_guard.py` ‚Äî functions following the `graph/progress_guard.py` convention. Track the session state machine on `LoopState`:

    IDLE ‚Üí LAUNCHED ‚Üí AWAITING_INPUT ‚Üí INPUT_SENT ‚Üí READING ‚Üí AWAITING_INPUT ‚Üí ‚Ä¶ ‚Üí EXITED ‚Üí VERIFIED ‚Üí COMPLETE

    Detect and emit recovery nudges (reuse the `recovery_kind` mechanism) for:

  • **Stalled session** ‚Äî `AWAITING_INPUT` for N turns with no input (Strategy B).
  • **Zombie session** ‚Äî process exited but final output never read.
  • **Orphan session** ‚Äî model started a new file/command without closing a prior session.
  • **Runaway input** ‚Äî model keeps sending to an already-exited process.

    • 6b. `task_complete` gate (extend the existing function)

    `tools/control_task_complete_gates.py::task_complete_gate_interactive_program(state)` (line 83) already blocks completion of interactive-program tasks that only ran a low-quality (score < 3) syntax/import verifier. Extend it to also:

  • Verify all interactive sessions on `LoopState` are closed (no zombies).
  • For `reactive`/`scripted` runs with ground-truth `required_interactions`, verify the transcript contains the expected outputs.

    8b. Scoring ‚Äî `aho/eval.py` must be extended (it currently ignores extra fields)

  • For exposed runs: assert the transcript shows each `send` followed by its `expect_output`.

    ## 10. Key Design Decisions

    1. **Interactivity is a task attribute, not a run-mode.** Reuse the `classify_task_mode` label + scratchpad visibility flag (`_expose_interactive_session_tools`) + FAMA `tool_policy` gating ‚Äî the same pattern already used for SSH session tools. Do **not** add an `"interactive"` run-mode to `ModeDecisionService.decide()`.

    2. **Local and remote tools are symmetric** ‚Äî only the transport differs (`interactive_start` vs `ssh_session_start`); both spawn via `shell.py::create_process` and drain via `read_stream_chunks`.

    3. **Exposed mode is thin** ‚Äî all orchestration is the model's; the harness only passes stdin/stdout through.

    4. **"Transparent" hides plumbing, not I/O.** The small model still sees prompts and supplies answers; it just never issues session tool calls. Scripted interactions use a single `interactive_run(command, inputs)` call (reliable); reactive interactions use a turn-bounded harness-driven state machine with a hard fallback to exposed tools.

    5. **`is_small_model_name` (‚â§14B) ‚Üí transparent; `is_over_twenty_b_model_name` (>20B) ‚Üí exposed; the 14‚Äì20B band defaults to exposed** and is overridable via `interactive_tool_policy`.

    6. **The runtime feature lives entirely in `src/smallctl/`.** `aho/` changes are limited to eval (challenges + `eval.py` scoring); no production logic depends on `aho/` interceptors.

    7. **Snapshots are the contract, not stdin streaming.** The model never sees a raw live stream ‚Äî it sees discrete snapshots (`new_output` + `status` + `detected_prompt`) from a read. This is what makes menus, log floods, progress bars, and "waiting" uniformly representable, and what lets the small-model path compact/throttle output before it reaches the prompt.

    8. **No session may run unbounded.** Three timers (per-read `deadline_sec`, `idle_grace_sec`, wall-clock `timeout_sec`) plus reactive-loop send/turn ceilings. A `timed_out`/`idle` session is a failure, not a completion (enforced at the `task_complete` gate). This fixes a real bug: the existing SSH module stores `timeout_sec` but never enforces it.
