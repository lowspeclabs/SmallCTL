Working Diagnosis

The addendum points to four separate problem classes that should not be collapsed:

  1. Benchmark/reporting is losing or misrepresenting evidence. aho/run_baseline.py currently discovers log dirs from the first logging_ready path, but RunLogger.set_session_id() can rename that directory later. It also counts only a narrow subset of events and renders duplicate challenge-NN headings across sysadmin and coding tasks.
  2. Many failures are timeout-after-progress loops. The runtime often has enough evidence, has written or is about to write the required artifact, or has already verified a coding change, but the model keeps reading, rechecking, or polling loop_status.
  3. Some failures are local/remote routing drift. Coding tasks that target local ./temp/*.py files are still exposed to, or prompted toward, SSH tools through stale handoff/memory signals and broad network profiles.
  4. Some failures are backend/model-stream failures rather than task-control failures. At least one run hit a remote protocol stream failure, and at least one emitted initial assistant text without any completed tool dispatch before the run timeout.

The repair should therefore split instrumentation fixes from runtime behavior changes. Otherwise the next benchmark will still mix reporting bugs, backend instability, and genuine agent-control regressions.

Phase 1 - Fix Benchmark Report Ground Truth

1. Persist Actual Log Directory

Problem:

  • src/smallctl/main.py prints {"status": "logging_ready", "run_log_dir": ...} before the logger may be renamed by RunLogger.set_session_id().
  • aho/run_baseline.py:discover_log_dir() trusts that early path if it exists.
  • The markdown report can link a stale prefix, e.g. timestamp suffix matches but run-id prefix does not.

Implementation:

  • Add a stable machine-readable event after session binding that records the final run_logger.run_dir.
  • Prefer writing this into both stdout and harness logs:
  • stdout event: {"status":"log_dir_finalized","run_log_dir":"...","reported_run_log_dir":"..."}
  • harness event: log_dir_finalized with the same fields.
  • In RunLogger.set_session_id(), after rename, log or expose enough state for the caller to emit the finalized directory.
  • In aho/run_baseline.py, replace discover_log_dir(stdout) with discover_log_dirs(stdout) returning:
  • reported_log_dir: first logging_ready path.
  • actual_log_dir: latest log_dir_finalized path, or a timestamp-suffix search fallback under .venv/logs / logs.
  • Store both on each challenge result as reported_log_dir and actual_log_dir; render actual_log_dir as the main Log dir.

Acceptance checks:

  • Add a unit test where stdout contains logging_ready for 39da4dbd-20260525-130259 and log_dir_finalized for 1d2d5b35-20260525-130259; report uses the latter.
  • Add a fallback test where only a stale prefix exists in stdout but a unique on-disk directory with the same timestamp suffix exists.

2. Rebuild Report Metrics From Logs

Problem:

  • review_logs() counts ask_human only from tools.jsonl dispatch_start, tool failures only from dispatch_complete success=false, and guards only from a small event-name subset.
  • The addendum confirms 03bccce4-20260525-125210/tools.jsonl has a successful ask_human, while the markdown showed AskHuman=0.
  • task_summary.json and full logs disagree with the markdown in several places.

Implementation:

  • Introduce a dedicated parser module, for example aho/log_review.py, instead of growing aho/run_baseline.py.
  • Parse these inputs when present:
  • task_summary.json
  • harness.jsonl
  • tools.jsonl
  • chat.jsonl
  • model_output.jsonl
  • text fallbacks: harness.log, tools.log, model_output.log
  • Count ask_human from both dispatch_start and dispatch_complete records, deduped by operation/tool-call id where available.
  • Count tool failures from every failed dispatch result, failed tool envelope, and tool error event, not only one event shape.
  • Count guard activity from event names and metadata:
  • event contains guard, blocked, loop, fama, verifier, recovery, stall, or task_complete_*_loop_status
  • result metadata contains active_mitigation, reason, error_kind, or recovery_kind
  • Preserve top examples for each count so the report can explain the number.

Acceptance checks:

  • Fixture for 03bccce4-20260525-125210 yields ask_human_count >= 1.
  • Fixture with failed ssh_exec, wrong-path file operation, and shell quoting error yields nonzero tool_failure_count.
  • Existing test_logwatch.py and new aho/tests/test_log_review.py cover event-shape variants.

3. Make Report Rows and Headings Unambiguous

Problem:

  • Sysadmin and coding challenges both render as challenge-03, making headings ambiguous.
  • Completion status alone hides unverified completions and backend failures.

Implementation:

  • Carry category through run_challenge() results as category: sysadmin|coding.
  • Render challenge keys as sysadmin/challenge-03 and coding/challenge-03 in tables and headings.
  • Add columns:
  • FinalClass
  • Verified
  • Backend
  • LogDir
  • Split status into:
  • completed
  • verified_completed
  • needs_human
  • backend_failed
  • timeout_after_progress
  • timeout_no_first_tool
  • tool_loop_timeout
  • failed
  • crashed
  • Treat challenge_progress.verified_after_last_change=false as a quality warning even when final status is completed.

Acceptance checks:

  • Markdown contains no duplicate ### challenge-03 headings.
  • A completed coding run with verified_after_last_change=false is not counted as verified_completed.
  • 1d2d5b35-20260525-130259 classifies as backend_failed, not timeout spiral.
  • d8eab62b-20260525-134642 classifies as timeout_no_first_tool or model_pre_tool_stall.

Phase 2 - Add Terminal-State Breakers

4. Completion Decision After Evidence and Artifact

Problem:

  • Sysadmin runs such as 23e96926, 33ef4951, and 8240234a gathered enough evidence or were ready to write reports, then kept checking until timeout.
  • The existing prompt says to call task_complete, but there is no hard terminal policy for "required artifact exists + post-write read/verification done + no failing verifier".

Implementation:

  • Add a terminal-state detector, likely in src/smallctl/challenge_progress.py or a new src/smallctl/task_completion_policy.py.
  • It should evaluate:
  • required output path exists, either from task target extraction or write-session target.
  • a successful write was followed by one read/verification.
  • current verifier is absent, passing, or explicitly waived.
  • no active write session requires a next section or finalization.
  • When true, suppress repeated exploratory read/list/status tools for one turn and inject a strong nudge:
  • "The required artifact exists and verification is complete. Next action must be task_complete or task_fail with a concrete blocker."
  • If the same condition persists after one nudge, block nonterminal read-only tools with a ToolEnvelope explaining the required terminal decision.

Likely integration points:

  • src/smallctl/graph/chat_progress.py: nudge text for repeated no-action/summary loops.
  • src/smallctl/graph/tool_loop_guards.py: block repeated read/status calls after terminal readiness.
  • src/smallctl/tools/dispatcher.py: last-resort block before dispatch.

Acceptance checks:

  • Synthetic state with written ./temp/sudo_audit.txt, successful file_read, and no failing verifier blocks another dir_list or loop_status and requires task_complete.
  • Coding state with verified_after_last_change=true blocks redundant tests/read loops.
  • State with an open write session does not trigger terminal completion.

5. Same-Conclusion Repetition Breaker

Problem:

  • Logs show repeated conclusions equivalent to "I have enough evidence", "now write the report", and "call task_complete" without the actual terminal action.

Implementation:

  • Track normalized assistant-thought/assistant-text conclusion signatures in GraphRunState or state.scratchpad.
  • Detect repetitions of phrases in these families:
  • enough evidence / sufficient evidence / all evidence gathered
  • now write report / prepare report / create artifact
  • final answer ready / call task_complete / complete task
  • After N repeats, inject a recovery message requiring the concrete next action:
  • if artifact missing: file_write
  • if artifact present and verified: task_complete
  • if verifier failed: one focused repair or task_fail
  • Keep this separate from generic repeated-thinking detection so the report can count it as terminal_state_breaker_count.

Acceptance checks:

  • Two repeated "enough evidence" turns with no tool call cause a terminal-state nudge.
  • If the required artifact is missing, the nudge points to file_write, not task_complete.

6. Write-Session Deadline

Problem:

  • Later coding runs repeatedly wrote or repaired script sections but did not converge within the 300s cap.
  • Current write-session guidance is detailed, but the model can still keep patching or rechecking after a successful artifact write and one verifier/read.

Implementation:

  • Extend src/smallctl/challenge_progress.py:
  • track successful_artifact_write_step
  • track first_post_write_verification_step
  • track nonterminal_steps_after_verified_write
  • After write + verification/read, require task_complete unless:
  • a verifier explicitly fails after the last change.
  • a required output path is still missing.
  • a write session is still open and finalizable.
  • Report this as write_deadline_terminal_guard.

Acceptance checks:

  • After file_write of ./temp/x.py and passing unittest, another exploratory file_read is blocked with terminal guidance.
  • After a failing unittest, the guard allows a focused patch.

Phase 3 - Fix Local/Remote Routing Drift

7. Hard Local Coding Tool Scope

Problem:

  • Local coding tasks targeting ./temp/*.py were routed through SSH guidance or attempted ssh_file_write / ssh_exec.
  • Existing remote_scope_is_active() returns false when explicit local targets are present, but tools can still be exposed via profiles, retry exposures, stale handoffs, or prompt guidance.

Implementation:

  • Add a single predicate, for example task_is_local_coding_target(state, task), based on:
  • task_mode == local_execute
  • challenge_progress.task_category == coding
  • required output path starts with ./temp/ and ends with .py
  • no explicit remote host/IP/SSH target in the current raw task
  • Use it to suppress SSH tools in filter_tools_for_runtime_state():
  • hide ssh_exec
  • hide ssh_file_read
  • hide ssh_file_write
  • hide ssh_file_patch
  • hide ssh_file_replace_between
  • Also prevent _append_retry_tool_exposures() from re-adding SSH tools for local coding tasks.
  • Add a prompt line in src/smallctl/prompts.py only when this predicate is true:
  • "LOCAL CODING TASK: use local file tools and shell_exec only; SSH tools are out of scope for this task."

Acceptance checks:

  • A task that says "Build a self-contained Python script at ./temp/foo.py" exposes no SSH tools even if network profile is active.
  • A sysadmin task that says "SSH to host ... write local report ./temp/foo.txt" still exposes SSH tools and local file_write.
  • A local coding task with stale _last_task_handoff remote metadata does not re-enable SSH tools.

8. Memory Retrieval Safety For Local Coding

Problem:

  • Stale memories like requested_ssh_file_patch appear to leak into local script generation.
  • memory_namespace.py already scopes namespaces by task mode, but retrieved memories and working-memory next actions can still bias prompts.

Implementation:

  • For local coding tasks, filter retrieved experience memories that contain remote-only tool intent unless the current task has an explicit remote target.
  • Filter or demote next actions containing ssh_exec, ssh_file_*, or requested_ssh_file_patch before rendering PromptStateFrame.
  • Add a runlog event when memory is suppressed:
  • remote_memory_suppressed_for_local_task
  • include memory id, namespace, and reason.

Acceptance checks:

  • Test in tests/test_task_intent_retrieval_tags.py or a new prompt-frame test: local ./temp/*.py task does not render requested_ssh_file_patch.
  • Remote sysadmin task still renders relevant SSH memory.

9. Sysadmin Local Artifact Clarity

Problem:

  • Sysadmin tasks need remote evidence collection but local report writes. The prompt contains guidance, but runtime/reporting should make this explicit.

Implementation:

  • When task_mode == remote_execute and task target paths include ./temp/.txt or ./temp/.md, record:
  • remote_evidence_required=true
  • local_artifact_paths=[...]
  • Render this in the prompt state frame:
  • "Remote evidence collection, local artifact write."
  • In tool guards, allow file_write for those local artifacts even when remote scope is active.
  • If the model attempts ssh_file_write to a local artifact path, block with recovery action: use local file_write(path='./temp/...').

Acceptance checks:

  • Sysadmin challenge can use ssh_exec for evidence and file_write for ./temp/*.txt.
  • ssh_file_write(path='./temp/foo.txt') is blocked with an explicit local artifact correction.

Phase 4 - Improve Tool API Recovery

10. File Write Error Messages

Problem:

  • Logs show repeated wrong recovery after file-write failures, including existing-file and chunk-session confusion.

Implementation:

  • Audit direct file_write, write-session file_write, file_patch, and ast_patch failure messages.
  • Ensure every failed write result has:
  • error_kind
  • next_required_tool
  • exact required fields
  • exact recovery action
  • Add messages for:
  • file exists: "use overwrite/replace mode" if that is the intended operation.
  • active write session: "continue session with write_session_id and next_section_name."
  • patch-existing first chunk: "use replace_strategy='overwrite' or narrow file_patch / ast_patch."

Acceptance checks:

  • Unit tests assert exact recovery hints for existing-file, active-session, and patch-existing first-chunk failures.

11. Expected Empty Results For SSH/Shell Probes

Problem:

  • SSH command exits from grep, find, which, package probes, and status checks can be valid negative evidence but are classified as generic failures.

Implementation:

  • Add a normalization helper in src/smallctl/tools/shell_support.py or src/smallctl/graph/shell_outcomes.py:
  • classify expected miss commands as empty_result when stdout/stderr semantics are clear.
  • preserve nonzero exit code in metadata.
  • mark result success for absence probes only when command shape is recognized.
  • Extend prompt guidance and report parser to count empty_result separately from failures.

Acceptance checks:

  • grep -q pattern missing-file remains failure.
  • find /path -name thing -print -quit with no output can be empty result when used as an absence check.
  • which missing_binary can be empty result when the task asks whether it exists.

12. Safer SSH Command Quoting

Problem:

  • The addendum notes quoting errors and large inline SSH commands.

Implementation:

  • Add a small command composition helper for generated benchmark sysadmin prompts or recovery guidance:
  • prefer remote script upload plus bounded execution for long multi-line logic.
  • prefer shlex.quote() for interpolated remote paths and patterns.
  • Add guidance in prompts.py for remote report probes:
  • batch read-only checks, but avoid deeply nested quoting.
  • write a temporary remote script only when commands exceed a safe complexity threshold.

Acceptance checks:

  • Tests cover generated SSH commands with paths containing spaces and quotes.
  • Tool failures caused by shell quote parse errors are classified distinctly.

Phase 5 - Backend Failure Classification

13. Retry Incomplete Chunked Reads Once

Problem:

  • 1d2d5b35-20260525-130259 failed with httpx.RemoteProtocolError, which is not a model-control failure.
  • Current stream handling has chunk-error fallback and stream-ended handling, but incomplete body/protocol failures need a separate benchmark class.

Implementation:

  • In the client stream path, catch httpx.RemoteProtocolError and equivalent incomplete-read exceptions.
  • Retry once with the same conversation state if no tool dispatch has occurred for that model call.
  • If retry fails, set final result:
  • status=failed
  • error_type=backend_stream_failure
  • details include provider profile, attempt count, and exception type.
  • Report this as backend_failed, not timeout_after_progress.

Acceptance checks:

  • Unit test simulates a remote protocol error on first stream and success on retry.
  • Unit test simulates two failures and verifies backend_stream_failure.

14. Pre-Tool Stall Classification

Problem:

  • d8eab62b-20260525-134642 emitted initial assistant text but no completed tool dispatch before timeout.

Implementation:

  • Track per-run:
  • first assistant token timestamp
  • first assistant text timestamp
  • first tool dispatch start timestamp
  • first tool dispatch complete timestamp
  • On timeout, classify:
  • timeout_no_first_tool: no dispatch start.
  • timeout_pre_tool_after_text: assistant text exists, no dispatch complete.
  • timeout_after_progress: at least one successful tool dispatch or artifact/write progress.
  • tool_loop_timeout: repeated similar failed/read-only calls dominate the tail.
  • Store these in task_summary.json and the AHO report.

Acceptance checks:

  • Synthetic logs with assistant text and no tool dispatch classify as timeout_pre_tool_after_text.
  • Synthetic logs with successful tools and no completion classify as timeout_after_progress.

Phase 6 - Scoring and Experiment Design

15. Scoring Split

Implementation:

  • Replace the single success/failure summary in aho/run_baseline.py with:
  • completed count
  • verified completed count
  • needs human count
  • backend failed count
  • timeout after progress count
  • timeout no first tool count
  • tool loop timeout count
  • unverified completion warning count
  • Keep raw status in the per-run data so historical comparisons are possible.

Acceptance checks:

  • Report summary has separate verified completion and backend failure lines.
  • A completion with verified_after_last_change=false increments quality warnings.

16. Re-Run Order

Run experiments in this order so causes stay separable:

  1. Report-only patch:
  • log dir finalization
  • category headings
  • log-derived AskHuman/ToolFails/guard counts
  • new final classification
  1. Routing-only patch:
  • local coding suppresses SSH tools
  • remote memory suppression for local coding
  • sysadmin local artifact clarity
  1. Terminal-state breaker only:
  • required artifact exists + one verification/read
  • same-conclusion repetition breaker
  • write-session deadline
  1. Backend classification only:
  • one retry for incomplete stream
  • backend/pre-tool timeout classes
  1. Retrieval ablation:
  • disable retrieved memories for local coding tasks
  • compare remote/local drift against the routing-only patch

Each run should keep the 300s cap and the same 40-task input set so timeout movement is comparable to the baseline.

Success Criteria

  • Report rows link to actual log directories, not stale stdout prefixes.
  • AskHuman, ToolFails, and guard counts agree with tools.jsonl, harness.jsonl, and task_summary.json.
  • Sysadmin and coding challenge headings are unambiguous.
  • Backend failures are not counted as timeout spirals.
  • Local ./temp/*.py coding tasks do not see or use SSH tools unless the task explicitly names a remote host.
  • Sysadmin tasks can still collect remote evidence and write local ./temp/*.txt reports.
  • Timeout-after-progress count drops after terminal-state breakers without increasing premature unverified completions.
  • Verified completion is tracked separately from raw completion.
